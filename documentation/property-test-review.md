Reviewed all of it — the library (`src/ImageBuilder.Automation`), the model-based test harness (`ModelRepo`, `ModelRepoHost`, `World`, `AutomationGen`, `TreeDelta`, `FsTree`), and both property test files. Here's the assessment.

## The big-picture status, first

All nine tests in `RepoHostPropertyTests` run against `ModelRepoHost` — the in-memory executable spec — because `GitHubRepoHost` and `AzdoRepoHost` are still `NotImplementedException` stubs. So today these properties verify that the *specification is self-consistent*, not that any production code works. That's a legitimate design (pin down semantics first, then transfer via an equivalence test, which the class doc comment promises), but it means every one of these tests is only as necessary as the plan to write that equivalence test. If that plan ever gets dropped, this whole file becomes test code testing test code and should go. The five tests in `RemoteRepoUrlPropertyTests` do test real shipped code.

## RemoteRepoUrlPropertyTests

**`GitHubAuthenticatedCloneUrlEmbedsTokenLosslessly` / `AzdoAuthenticatedCloneUrlEmbedsTokenLosslessly`** — Property: for any repo identity and any token (including URI-hostile characters), the authenticated URL parses, the token round-trips through escaping, and the URL differs from `CloneUrl` only by the credential. **Good job, worth keeping.** The `Token` generator deliberately includes `@`, `:`, `/`, `%`, spaces, and non-ASCII — exactly where a naive string-interpolation bug would hide — and the assertions are direct. One gap: only `""` is tested for the empty case, never `null`, though `string.IsNullOrEmpty` in the implementation suggests null is meant to be legal. The GitHub/AzDO duplication is fine; it actually helps newcomers.

**`CloneUrlNeverContainsCredentials`** — Property: `CloneUrl.UserInfo` is empty for any names. **Doing a bad job, candidate to cut or demote.** It's near-tautological: `CloneUrl` never sees a token, and the `Name` generator excludes `@`, so there is no input that could ever make this fail. Randomization buys nothing. If you want the "clone URL is credential-free" guarantee documented, a one-line example test says it more honestly.

**`EmptyTokenYieldsBareCloneUrl`** and **`LocalRepoIgnoresToken`** — Properties: empty token → bare URL; `LocalRepo` ignores tokens entirely. **Both correct but mislabeled as properties.** The randomized inputs don't interact with the behavior under test (a single `if` branch / an unconditional ignore). These would be clearer to a property-testing newcomer as plain `[TestMethod]` example tests — keeping them in the property file teaches the wrong lesson about when randomization earns its keep.

## RepoHostPropertyTests

**`EnsureIsIdempotent`** — Property: running the same ensure twice changes nothing the second time. **Excellent.** Short, canonical, the defining reconciliation law. This is the test I'd show a newcomer first. Keep.

**`EnsurePullRequestConverges`** — Property: full postconditions of one ensure, case-split by outcome (Created / Updated / Unchanged / Stopped). **Necessary but the weakest on readability, and the most structurally suspect.** Two problems:
- At ~125 lines with a nested three-way split inside `Updated`, it's the file's wall. Splitting into four properties (`EnsureCreatesPullRequestCorrectly`, `EnsureStopsOnForeignCommits`, …) that each `return` early when the outcome doesn't match would cost a little generator efficiency and buy a lot of comprehension.
- Its assertions mirror the model's control flow almost branch-for-branch (`Append` → base is head tree, etc.). Since the model is also test code, a shared misunderstanding passes both sides silently. This is inherent to model-based testing and acceptable, but it's why the equivalence test against real hosts is load-bearing — worth a comment saying so.

**`EnsureBranchConverges`** — Property: branch tip becomes delta-applied-to-old-tip via fast-forward, or nothing happens when already converged. **Good.** Small, complete, readable. Keep.

**`EnsureOutcomeIsHonest`** — Property: result metadata never lies (Unchanged ⇒ bit-identical world, Stopped ⇒ no ref moved, `CommitSha` is a genuinely new branch tip, Created/Updated match prior PR existence). **Good, keep.** It overlaps `EnsurePullRequestConverges` substantially, but it covers both command types uniformly, it's the most readable statement of the contract in the file, and the "CommitSha is *new*" check (`preTips.Values.ShouldNotContain`) is asserted nowhere else. If you ever trim for overlap, trim the other one's redundant clauses, not this.

**`DryRunDoesNotMutateAndPredictsRealOutcome`** — Property: dry run is the identity on remote state and its outcome maps deterministically onto what a real run would have done. **Excellent.** Clean metamorphic test, the fork-and-compare pattern is easy to follow. Keep.

**`HumanWorkIsNeverDestroyed`** — Property: under `CommentAndStop`, across arbitrary interleavings of automation runs and human pushes/closes/merges, no human commit ever becomes unreachable; plus no duplicate stop-comments. **The most valuable test in the repo** — it's the safety claim that justifies the library's existence, and the stateful-sequence shape is exactly what property testing is for. Keep unconditionally. Two nits: it bundles a second, unrelated property (comment dedup) — split it out so each failure names one broken promise; and `iter: 1000` is an unexplained magic number — a one-line comment on why the default wasn't enough would help.

**`UpdateStrategiesAgreeWhenNoPullRequestExists`** — Property: Replace and Append are indistinguishable on the create path. **Marginal; my candidate to cut.** Three strikes: (1) the early-`return` precondition filter is a silent-vacuity trap — nothing tells you what fraction of generated worlds actually exercise the property, and a newcomer copying this pattern can write a test that passes while asserting nothing; (2) the behavior it pins is a single line in the model (`Append` only consulted in `UpdateAsync`); (3) `EnsurePullRequestConverges`' Created case already asserts the create path ignores strategy implicitly. If you keep it, generate no-PR worlds directly instead of filtering.

**`EnsuresWithDistinctKeysCommute`** — Property: ensures with distinct keys commute (same final world either order). **Conceptually nice, but it has a transfer problem worth flagging now.** It only holds because the model assigns PR URLs per-head-branch (`Repo.PullRequests.Count(pr => pr.HeadBranch == spec.Key)` in `ModelRepoHost.cs:135`) — that numbering scheme looks like it was chosen *to make this property pass*. Real GitHub/AzDO PR numbers are globally sequential, so real hosts will never satisfy snapshot equality including URLs, and the promised equivalence test can't transfer this property as written. Either compare snapshots modulo URL, or drop the test and accept commutativity as a corollary of key-independence. As-is, it's quietly testing an artifact of the model.

**`ApplyRunsOnCleanCheckoutOfBranchTip`** — Property: the user's `ApplyChanges` callback only ever sees a clean checkout of exactly the right base tree. **Good and genuinely necessary** — it's the contract from the caller's side, tested by instrumenting the callback, which nothing else covers. It also pins a real spec decision (Apply runs even when the result is Stopped, via `ShouldNotBeEmpty`). Keep; maybe add one sentence in the doc comment noting that the non-empty assertion is deliberate, since that's easy to read as incidental.

## On the newcomer-readability goal specifically

The doc comments are genuinely excellent — every test states its property in prose, and the `ToString` overrides on `Setup`/`TreeDelta` mean shrunk counterexamples will actually be legible. Three things would move the needle most:

1. **The real comprehension cost isn't the properties — it's the ~450 lines of model** (`ModelRepo` + `ModelRepoHost` + `World`). A newcomer can't judge whether a test is right without trusting the model. A short README in the test project explaining the architecture — "model-based testing: properties run against an executable spec; an equivalence test will transfer them to real hosts" — with a diagram would do more than any per-test edit.
2. **The bitmask generators in `AutomationGen` (`BuildTree`/`BuildDelta`) are the most cryptic code in the suite.** `(mask >> i & 1) != 0` over a path array is a clever encoding, but `from include in Gen.Bool` per path (or a subset generator) says the same thing in the vocabulary the rest of the file uses. Newcomers will read the generators before the properties.
3. **Cut the tests that randomize over inputs the behavior doesn't depend on** (`CloneUrlNeverContainsCredentials`, and demote `EmptyTokenYieldsBareCloneUrl`/`LocalRepoIgnoresToken` to example tests). The fastest way to make property tests approachable is for every remaining one to visibly *need* to be a property test.

Net recommendation: keep 6 of the 9 host properties as-is or split for readability (`EnsureIsIdempotent`, `EnsurePullRequestConverges`, `EnsureBranchConverges`, `EnsureOutcomeIsHonest`, `DryRun…`, `HumanWorkIsNeverDestroyed`, `ApplyRunsOnCleanCheckout…` — that's 7, actually), cut or rework `UpdateStrategiesAgreeWhenNoPullRequestExists` and `EnsuresWithDistinctKeysCommute`, keep the two token-escaping URL properties, and demote the other three URL tests to plain examples. Happy to make any of these changes if you want.
